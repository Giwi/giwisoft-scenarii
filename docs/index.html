Scenarii

YAML-defined web test scenarios - run periodically via a headless browser or native HTTP, stored in SQLite, monitored via a modern Dashboard.

GitHub Packages CI

What is Scenarii?

Scenarii lets you write web test scenarios as simple YAML files, run them on a cron schedule via a headless browser or native HTTP, and track results over time on a dashboard. Get alerted on failures via Telegram, Slack, Discord, email, or a generic webhook.

YAML-defined

Scenarios are plain YAML files - no code. Define HTTP requests or browser interactions with step-by-step expectations.

Cron scheduling

Run every 5 minutes, hourly, daily - any cron expression. Built-in scheduler with pause/resume from the dashboard.

SQLite storage

All metrics persisted in SQLite with WAL mode. Configurable retention and optional automated daily backups.

Dashboard

Full dashboard with dark/light mode, response time charts, success rate, SLA gauge, run history, JSON/CSV export, and public status pages.

Multi-channel alerts

State transition notifications (pass -> fail, fail -> pass) via Telegram, Slack, Discord, email, or custom webhook with retry logic.

Browser + HTTP

Native fetch for API monitoring (no browser needed) or Lightpanda headless browser for full page interaction with retry and screenshots.

All features

CLI & API

Commands: validate, trigger, server, status, config. Full REST API with 20+ endpoints.

Authentication

Optional OIDC dashboard auth with HTTP-only session cookies. API key auth for Prometheus metrics endpoint.

Notifications

Telegram, Slack, Discord, email (Mailgun), generic webhook. State transition triggers with retry and exponential backoff.

Prometheus metrics

Native Prometheus/OpenMetrics export: run count, duration, success rate, step-level metrics, notification delivery stats.

SQLite storage

WAL mode for concurrent reads. Configurable retention (default 7 days). Optional automated daily backups. No external database.

Container

Multi-arch images (linux/amd64 + arm64). HEALTHCHECK, non-root user, dumb-init. Published to GitHub Container Registry on each push.

Hot-reload

Settings YAML watched and reloaded automatically. Scenario directory scanned every 5s - add, remove, or rename .yml files without restarting.

Pause / Resume

Toggle individual schedules on and off from the dashboard without deleting YAML files.

HTTP expectations

Check status, status_in, body_contains, body_matches, headers, JSON path, JSON value, and response_time_under.

Browser actions

Navigate, click, fill, type, select, check, uncheck, evaluate JS, scroll, wait for elements, screenshot, screenshot comparison against baseline.

All HTTP methods

GET, POST, PUT, PATCH, DELETE - all standard methods. Custom headers, JSON body, form data. Native fetch, no Playwright needed.

Testing

Node built-in test runner. Tests cover execution queue, notification state machine, URL resolution, variable interpolation, YAML parsing, and more.

Quick start

npm

git clone https://github.com/Giwi/giwisoft-scenarii.git
cd giwisoft-scenarii
npm install
npm run build

# Start the server (loads scenarios from ./scenarios, serves dashboard)
node dist/index.js server

# Or run a scenario once
node dist/index.js --once scenarios/example.yml

Open http://localhost:3000 for the dashboard.

Docker

docker pull ghcr.io/giwi/giwisoft-scenarii:latest

mkdir -p scenarios db
cp settings.example.yaml settings.yml

docker run -d \
  --name scenarii \
  -p 3000:3000 \
  -v $(pwd)/scenarios:/scenarios \
  -v $(pwd)/db:/app/db \
  -v $(pwd)/settings.yml:/app/settings.yml:ro \
  ghcr.io/giwi/giwisoft-scenarii:latest

Sample scenarios

Scenarios are pure YAML - define steps, expectations, extraction variables, and optional conditions. No programming required.

HTTP API health check

Basic HTTP scenario that verifies an API is responding with the expected status code and response time.

# scenarios/example.yml
name: API health check
description: Verify API health every 5 minutes
schedule: "*/5 * * * *"
base_url: https://jsonplaceholder.typicode.com

steps:
  - name: Get posts
    action: http.get
    url: /posts/1
    expect:
      status: 200
      response_time_under: 3000

  - name: Verify JSON content
    action: http.get
    url: /posts/1
    expect:
      status: 200
      json_path: $.userId      # check specific JSON field
      json_value: 1

  - name: Create a post
    action: http.post
    url: /posts
    body:
      title: foo
      body: bar
      userId: 1
    headers:
      Content-Type: application/json
    expect:
      status: 201

BROWSER Browser form test

Navigate to a page, wait for content, and evaluate JavaScript in the page - all through a headless Lightpanda browser.

# scenarios/browser-example.yml
name: Browser form test
description: Tests page load and content rendering
schedule: "*/10 * * * *"
base_url: https://example.com

steps:
  - name: Open page
    action: browser.navigate
    url: /

  - name: Verify page loaded
    action: browser.wait_for
    selector: body
    expect:
      has_text: "Example"

  - name: Check URL
    action: browser.evaluate
    script: "window.location.href"
    expect:
      url_contains: "example.com"

MIXED Browser + HTTP

Combine browser steps with HTTP API calls in the same scenario.

# scenarios/mixed.yml
name: Mixed browser and HTTP
base_url: https://example.com

steps:
  - name: Open page
    action: browser.navigate
    url: /
    expect:
      has_text: "Example"

  - name: Check external API
    action: http.get
    url: https://jsonplaceholder.typicode.com/posts/1
    expect:
      status: 200
      json_path: $.id
      json_value: 1

HTTP Conditional execution

Skip steps based on a previous step's result using the condition field. Extract variables with json_path and reuse with {{variable}}.

name: Conditional scenario
base_url: https://jsonplaceholder.typicode.com

steps:
  - name: Login
    action: http.post
    url: /auth/login
    body:
      username: admin
      password: secret
    expect:
      status: 200
    variables:
      token: $.access_token    # extract from response

  - name: Dashboard check
    action: http.get
    url: /api/dashboard
    condition:               # skip if Login failed
      if_step: "Login"
      if_success: true
    headers:
      Authorization: "Bearer {{token}}"
    expect:
      status: 200

BROWSER Full browser interactions

Fill forms, click buttons, select options, take screenshots, and compare against baselines.

name: Full browser test
base_url: https://example.com

steps:
  - name: Navigate to form
    action: browser.navigate
    url: /form

  - name: Fill field
    action: browser.fill
    selector: "#name"
    value: "Jane Doe"

  - name: Select country
    action: browser.select
    selector: "#country"
    value: "FR"

  - name: Check agreement
    action: browser.check
    selector: "#agree"

  - name: Submit
    action: browser.click
    selector: "#submit"

  - name: Wait for result
    action: browser.wait_for
    selector: ".success-message"
    timeout: 5000
    expect:
      has_text: "Thank you"

  - name: Screenshot
    action: browser.screenshot
    value: /tmp/screenshot.png

HTTP Variable extraction & reuse

Extract values from JSON responses and reference them in later steps.

name: Variable extraction demo
base_url: https://jsonplaceholder.typicode.com

steps:
  - name: Get first post
    action: http.get
    url: /posts/1
    expect:
      status: 200
    variables:
      postTitle: $.title       # extracted from response
      postId: $.id

  - name: Get user for post
    action: http.get
    url: /users/{{postId}}
    expect:
      status: 200

  - name: Verify content
    action: http.get
    url: /posts/{{postId}}
    expect:
      body_contains: "{{postTitle}}"

Actions Reference

Every step must have a name and an action. Actions are either HTTP-based (no browser needed) or browser-based (Lightpanda headless browser).

HTTP Actions

HTTP steps use the native fetch API - no Playwright or browser required for simple API monitoring.

ActionRequired fieldsOptional fields
http.geturlheaders, expect
http.posturlheaders, body, expect
http.puturlheaders, body, expect
http.patchurlheaders, body, expect
http.deleteurlheaders, expect

HTTP expectations

ExpectationDescription
status: 200Exact HTTP status code match.
status_in: [200, 201]Status code is one of the listed values.
body_contains: "text"Response body contains the given string.
body_matches: "regex"Response body matches the given regex.
header_contains: "Content-Type: json"Response header contains the given value (format: "HeaderName: value").
header_matches: "X-RateLimit: \\d+"Response header matches the given regex (format: "HeaderName: regex").
json_path: "$.user.id"Extract a JSON path and optionally check json_value.
json_value: 42Value at the json_path must match (used together).
response_time_under: 3000Response time must be under the given milliseconds.

BROWSER Actions

Browser steps run through a headless Lightpanda instance via Playwright CDP. Steps automatically retry up to 2 times with exponential backoff (1s, 2s) on failure.

ActionFieldsDescription
browser.navigateurlNavigate to a URL. Falls back to domcontentloaded if the load event does not fire.
browser.clickselectorClick an element. Uses Playwright actionability checks (visible, enabled, stable, in viewport).
browser.fillselector, valueFill an input field using native property setters for reliable React/Angular form control compatibility.
browser.typeselector, valueType into an input field. Like fill but dispatches input + change events.
browser.selectselector, valueSelect an option from a <select> element.
browser.checkselectorCheck a checkbox or radio button.
browser.uncheckselectorUncheck a checkbox.
browser.wait_forselector, timeout, expectWait for an element to appear. Optional expect can check has_text, not_has_text, or url_contains.
browser.evaluatescriptRun arbitrary JavaScript in the page context. Useful for programmatic clicks, data extraction, and complex interactions.
browser.scrollvalue or selectorScroll value pixels down, or scroll a selector into view. Useful for lazy-loaded content or elements below the fold.
browser.screenshotvalue (path)Take a full-page screenshot and save to the given path.
browser.screenshot_comparevalue (baseline path)Take a screenshot and compare file size against a baseline. Creates baseline on first run; reports diff percentage on subsequent runs.

Step options

FieldTypeDescription
timeoutnumberPer-step timeout in milliseconds (overrides the default 30s for browser actions, 10s for HTTP).
conditionobjectConditional execution: { if_step: "Step name", if_success: true } skips this step unless the named step succeeded (or failed, if false).
variablesobjectExtract values from HTTP JSON responses: { varName: "$.json.path" }. Referenced as {{varName}} in later steps.
expectobjectAssertions on the step result (status code, body content, headers, JSON path, response time).

Notifications

Get alerted when a scenario fails and when it recovers. Create a settings.yaml file to configure channels.

Supported channels

Telegram

Bot token + chat ID. Configure TELEGRAM_BOT_TOKEN and TELEGRAM_CHAT_ID env vars or settings.yaml.

Email (Mailgun)

Mailgun API key and domain. Supports multiple recipients. Includes a daily report sent at 8:00 AM.

Slack

Webhook URL. Simple JSON payload with scenario name, status, and duration.

Discord

Webhook URL. Embedded message with color-coded pass/fail status.

Generic webhook

Custom endpoint. Receives JSON payload for integration with any system.

Sample configuration

# settings.yaml
auth:
  enabled: true                  # optional dashboard auth
  oidc:
    issuer_url: "https://..."
    client_id: "YOUR_CLIENT_ID"
    client_secret: "YOUR_SECRET"

storage:
  retentionDays: 30              # data retention (default 7)
  backup:
    enabled: true                # automated daily backups
    cron: "0 4 * * *"

notifications:
  telegram:
    enabled: true
    bot_token: "YOUR_TOKEN"
    chat_id: "YOUR_CHAT_ID"
  email:
    enabled: true
    mailgun:
      api_key: "YOUR_KEY"
      domain: "your.domain"
      from: "scenarii <noreply@your.domain>"
    to:
      - "admin@example.com"
  slack:
    enabled: true
    webhook_url: "https://hooks.slack.com/..."

Per-scenario overrides

# settings.yaml -- override per scenario
scenarios:
  my-scenario:
    ignoreHTTPSErrors: true
    timeout: 60000
    notifications:
      enabled: false

Secrets can be overridden via environment variables: TELEGRAM_BOT_TOKEN, MAILGUN_API_KEY, etc.

Notifications trigger on state transitions (pass -> fail, fail -> pass) with retry logic (3 attempts, exponential backoff). The server watches settings.yaml and reloads it automatically on change.

Real-time updates

The server exposes a WebSocket endpoint at /ws that pushes live updates to connected clients.

Scenario run events

// Broadcast after every run

  "success": true,
  "duration_ms": 2500,

// Streamed during execution -- one per step
{

  "scenario_name": "Lusk.bzh validation",

The dashboard shows a live ticker in the detail toolbar while steps are executing - current step name, response time, and status icon update in real time.

Prometheus Metrics

The server exposes a /api/metrics endpoint in Prometheus/OpenMetrics format. Scrape it with Prometheus or any OpenMetrics-compatible collector.

# HELP scenarii_scenario_runs_total Total run count
# TYPE scenarii_scenario_runs_total counter
scenarii_scenario_runs_total{scenario="Lusk.bzh"} 142

# HELP scenarii_scenario_duration_ms Latest run duration
scenarii_scenario_duration_ms{scenario="Lusk.bzh"} 2540

# HELP scenarii_scenario_success Latest run result (1=pass, 0=fail)
scenarii_scenario_success{scenario="Lusk.bzh"} 1

# HELP scenarii_scenario_last_run_seconds Last run timestamp
scenarii_scenario_last_run_seconds{scenario="Lusk.bzh"} 1.756e9

# HELP scenarii_step_duration_ms Per-step duration
scenarii_step_duration_ms{scenario="Lusk.bzh",step="Homepage",action="http.get"} 127

# HELP scenarii_step_success Per-step success
scenarii_step_success{scenario="Lusk.bzh",step="Homepage",action="http.get"} 1

# HELP scenarii_notification_delivery_total Notification delivery count
scenarii_notification_delivery_total{status="success"} 12

Protect the endpoint with the api_key setting. Configure Prometheus to scrape http://localhost:3000/api/metrics with the key in the Authorization header.

# prometheus.yml
scrape_configs:
  - job_name: 'scenarii'
    authorization:
      credentials: 'YOUR_API_KEY'
    static_configs:
      - targets: ['localhost:3000']
        metrics_path: '/api/metrics'

CLI Reference

The scenarii CLI provides commands for running, validating, and managing scenarios.

# Start the API server + Dashboard
node dist/index.js server -p 3000 --db db/scenarii.db --scenarios-dir ./scenarios

# Validate a YAML file without running
node dist/index.js validate scenarios/example.yml

# Run a scenario immediately
node dist/index.js trigger scenarios/example.yml

# Show scheduled scenarios and storage status
node dist/index.js status

# Generate a settings.yaml template
node dist/index.js config --init
CommandDescription
serverStart HTTP server with Dashboard, scenario scheduler, and WebSocket. Scenarios directory is rescanned every 5s - add, remove, or rename .yml files without restarting.
validate <file>Parse a YAML file and validate its structure without executing it.
trigger <file>Run a scenario file immediately (single run, no scheduling).
statusList all scheduled scenarios and show database connectivity.
config --initGenerate a boilerplate settings.yaml with all notification channels commented out.
Server optionDefaultDescription
-p, --port3000HTTP listen port.
--dbdb/scenarii.dbPath to SQLite database file.
--scenarios-dir./scenariosDirectory containing .yml / .yaml scenario files. Watched every 5s for changes.
--settingsauto-detectPath to settings.yaml. Hot-reloaded on change.
