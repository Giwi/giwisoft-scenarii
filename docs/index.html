Scenarii

YAML-defined web test scenarios — run periodically via a headless browser or native HTTP, stored in SQLite, monitored via a modern Angular dashboard.

GitHub Container CI

What is Scenarii?

Scenarii lets you write web test scenarios as simple YAML files, run them on a cron schedule via a headless browser or native HTTP, and track results over time on a dashboard. Get alerted on failures via Telegram, Slack, Discord, email, or a generic webhook.

📝

YAML-defined

Scenarios are plain YAML files — no code to write. Define HTTP requests or browser interactions with step-by-step expectations.

⏰

Cron scheduling

Run scenarios every 5 minutes, hourly, daily — any cron expression. Built-in scheduler with pause/resume support.

📊

SQLite storage

All metrics are persisted in SQLite with WAL mode. Configurable retention and optional automated backups.

💻

Angular dashboard

Full-featured dashboard with dark/light theme, charts, SLA gauge, run history, JSON/CSV export, and public status pages.

🔔

Multi-channel alerts

Notify on state transitions (pass→fail, fail→pass) via Telegram, Slack, Discord, email (Mailgun), or a custom webhook.

🌐

Browser + HTTP

Use native fetch for API monitoring (no browser needed) or Lightpanda headless browser for full page interaction.

Quick start

# Clone and install
git clone https://github.com/Giwi/giwisoft-scenarii.git
cd giwisoft-scenarii
npm install
npm run build

# Start the server (loads scenarios from ./scenarios)
node dist/index.js server

# Or run a scenario once
node dist/index.js --once scenarios/example.yml

Open http://localhost:3000 for the dashboard.

Docker

docker pull ghcr.io/giwi/giwisoft-scenarii:latest

mkdir -p scenarios db
cp settings.example.yaml settings.yml

docker run -d \
  --name scenarii \
  -p 3000:3000 \
  -v $(pwd)/scenarios:/scenarios \
  -v $(pwd)/db:/app/db \
  -v $(pwd)/settings.yml:/app/settings.yml:ro \
  ghcr.io/giwi/giwisoft-scenarii:latest

Sample scenarios

Scenarios are pure YAML — define steps, expectations, extraction variables, and optional conditions.

HTTP API health check

Basic HTTP scenario that verifies an API is responding with the expected status code and response time.

name: API health check
description: Verify API health every 5 minutes
schedule: "*/5 * * * *"
base_url: https://jsonplaceholder.typicode.com

steps:
  - name: Get posts
    action: http.get
    url: /posts/1
    expect:
      status: 200
      response_time_under: 3000

  - name: Verify JSON content
    action: http.get
    url: /posts/1
    expect:
      status: 200
      json_path: $.userId
      json_value: 1

  - name: Create a post
    action: http.post
    url: /posts
    body:
      title: foo
      body: bar
      userId: 1
    headers:
      Content-Type: application/json
    expect:
      status: 201

BROWSER Browser form test

Navigate to a page, wait for content, and evaluate JavaScript in the page — all through a headless Lightpanda browser.

name: Browser form test
description: Tests a form submission flow in the browser
schedule: "*/10 * * * *"
base_url: https://example.com

steps:
  - name: Open page
    action: browser.navigate
    url: /

  - name: Verify page loaded
    action: browser.wait_for
    selector: body
    expect:
      has_text: "Example"

  - name: Check URL
    action: browser.evaluate
    script: "window.location.href"
    expect:
      url_contains: "example.com"

MIXED Browser + HTTP

Combine browser steps with HTTP API calls in the same scenario. Steps can reference results from previous ones.

name: Mixed browser and HTTP
base_url: https://example.com

steps:
  - name: Open page
    action: browser.navigate
    url: /
    expect:
      has_text: "Example"

  - name: Check external API
    action: http.get
    url: https://jsonplaceholder.typicode.com/posts/1
    expect:
      status: 200
      json_path: $.id
      json_value: 1

HTTP Conditional execution

Skip steps based on the success or failure of a previous step using the condition field.

name: Conditional scenario
base_url: https://jsonplaceholder.typicode.com

steps:
  - name: Login
    action: http.post
    url: /auth/login
    body:
      username: admin
      password: secret
    expect:
      status: 200
    variables:
      token: $.access_token

  - name: Dashboard check
    action: http.get
    url: /api/dashboard
    condition:
      if_step: "Login"
      if_success: true
    headers:
      Authorization: "Bearer {{token}}"
    expect:
      status: 200

BROWSER Browser interactions

Full browser interaction: fill forms, click buttons, select options, take screenshots, and compare against baselines.

name: Full browser test
base_url: https://example.com

steps:
  - name: Navigate to page
    action: browser.navigate
    url: /form

  - name: Fill form field
    action: browser.fill
    selector: "#name"
    value: "John Doe"

  - name: Select option
    action: browser.select
    selector: "#country"
    value: "FR"

  - name: Check checkbox
    action: browser.check
    selector: "#agree"

  - name: Submit
    action: browser.click
    selector: "#submit"

  - name: Wait for result
    action: browser.wait_for
    selector: ".success-message"
    timeout: 5000
    expect:
      has_text: "Thank you"

  - name: Take screenshot
    action: browser.screenshot
    value: /tmp/screenshot.png

HTTP Variable extraction

Extract values from JSON responses with json_path and reuse them in subsequent steps using {{variable}} syntax.

name: Variable extraction demo
base_url: https://jsonplaceholder.typicode.com

steps:
  - name: Get first post
    action: http.get
    url: /posts/1
    expect:
      status: 200
    variables:
      postTitle: $.title
      postId: $.id

  - name: Check user for post
    action: http.get
    url: /users/{{postId}}
    expect:
      status: 200

  - name: Final log
    action: http.get
    url: /posts/{{postId}}
    expect:
      body_contains: "{{postTitle}}"

Dashboard

The Angular frontend gives you real-time visibility into all your scenarios.

📋 Scenario list

Overview of all scenarios with pass/fail badges, tag filters, dependencies, and auto-refresh via WebSocket.

📈 Scenario detail

Response time trend chart, success rate chart, SLA gauge, step breakdown, paginated history with JSON/CSV export.

⏳ Live ticker

Inline step progress in the toolbar while a scenario is running — see which step is executing in real time.

🌐 Public status

Shareable per-scenario pages at /public/status/:name with charts — no authentication required.

🎯 SLA tracking

7-day SLA gauge per scenario with configurable window. Export SLA data via the detail page.

🛑 Real-time

WebSocket push for run completions and step progress. Manual refresh button on both list and detail pages.

Features

📦 CLI & API

Run via CLI (validate, trigger, server, status, config) or REST API. Rich API with 20+ endpoints.

🔐 Auth

Optional OIDC-based dashboard authentication with HTTP-only session cookies. API key auth for Prometheus metrics endpoint.

📡 Notifications

Telegram, Slack, Discord, email (Mailgun), or generic webhook. Fires on state transitions with retry logic and exponential backoff.

📰 Prometheus

Exposes metrics in Prometheus/OpenMetrics format: run count, duration, success rate, step-level metrics, and notification delivery stats.

📊 SQLite storage

WAL mode for concurrent reads. Configurable retention. Optional automated daily backups. No external database needed.

🐳 Container

Multi-arch Docker images (linux/amd64 + arm64). HEALTHCHECK, non-root user, dumb-init. Published to GitHub Container Registry.

🧱 Hot-reload

Settings YAML is watched and reloaded automatically. Add or modify scenario files on disk and they're picked up on the next schedule.

🏗️ Pause/Resume

Toggle individual scheduled scenarios on and off from the dashboard without deleting their YAML files.

🔧 API expectations

Check status code, status_in list, body content (contains/matches regex), headers, JSON path values, and response time thresholds.

🖥️ Browser actions

Navigate, click, fill, type, select, check, evaluate JS, wait for elements, take screenshots, and compare against baselines.

📜 HTPass

HTTP, POST, PUT, PATCH, DELETE — all standard methods supported. Custom headers, JSON body, form data.

✅ Testing

Node built-in test runner. Tests cover execution queue, notification state machine, URL resolution, variable interpolation, YAML parse/serialize.

API endpoints

# Scenarios
GET    /api/scenarios                   # List all scenarios (?tag=critical)
GET    /api/scenarios/:name             # Detail with paginated history
GET    /api/scenarios/:name/history     # Raw run history
POST   /api/scenarios/:name/run         # Trigger ad-hoc run
POST   /api/scenarios/:name/cancel      # Abort running scenario
POST   /api/scenarios/:name/pause       # Pause scheduled scenario
POST   /api/scenarios/:name/resume      # Resume paused scenario
GET    /api/scenarios/:name/config      # Download YAML config
PUT    /api/scenarios/:name/config      # Save/update YAML config
DELETE /api/scenarios/:name/config      # Delete scenario file
GET    /api/scenarios/:name/export/json # Download history as JSON
GET    /api/scenarios/:name/export/csv  # Download history as CSV
GET    /api/scenarios/:name/sla         # SLA stats (?days=7)

# Auth
GET    /api/auth/login                  # OIDC login redirect
GET    /api/auth/callback               # OIDC callback
GET    /api/auth/me                     # Session status
POST   /api/auth/logout                 # Clear session

# Public / system
GET    /api/public/scenario/:name       # Public per-scenario detail
GET    /api/status                      # Public summary
GET    /api/health                      # Health check
POST   /api/backup                      # Manual DB backup
GET    /api/tags                        # List tags
GET    /api/metrics                     # Prometheus format
